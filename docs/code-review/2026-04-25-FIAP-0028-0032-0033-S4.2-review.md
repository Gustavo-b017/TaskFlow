## Pontos Fortes

### Arquitetura
- `taskStorage.ts` segue o princípio de responsabilidade única (SRP): é pura I/O sem estado, sem hooks, sem lógica de contexto. Pode ser testado e substituído de forma isolada.
- A separação `services/ ← context/` está correta. `TaskContext` não faz I/O direto — delega ao serviço.
- O padrão `tasksRef` (useRef) é correto para capturar o estado mais recente em chamadas consecutivas sem re-render, evitando closures sobre valores desatualizados.
- A interface `TaskContextData` é exportada, permitindo tipagem dos consumidores sem acoplamento direto ao contexto.

### Rollback Otimista
- O mecanismo de rollback (capturar `previous`, aplicar no catch) é idiomático e garante que a UI reverte o estado visual caso `saveTasks` falhe.
- A tríade `setTasks(updated) → await saveTasks → catch → setTasks(previous)` está implementada consistentemente nos três mutadores.

### Segurança de Tipos
- Zero uso de `any` nos arquivos revisados.
- A assinatura `Omit<Partial<Task>, 'id' | 'createdAt' | 'updatedAt'>` em `updateTask` previne que consumidores sobrescrevam campos gerenciados pelo contexto.

### Cobertura de Testes
- 29 testes (6 taskStorage, 21 TaskContext, 2 useTasks), todos passando.
- Testes de rollback cobrem falha do saveTasks nos três mutadores.
- Uso de `jest.useFakeTimers` para asserções de `updatedAt` é a prática correta.
- Testes de validação cobrem título vazio e título com apenas espaços.

## Issues Encontrados e Corrigidos

### Crítico 1 — Dead catch em TaskContext.init() (CORRIGIDO)

**Arquivo:** `src/context/TaskContext.tsx` — bloco `useEffect`

**Problema:** O `catch {}` em `init()` era código morto. `loadTasks()` nunca lança (suprime erros internamente), tornando o bloco nunca alcançável. Código morto mascara intenção e é armadilha para manutenção futura.

**Correção aplicada:** Adicionado comentário documentando que o catch é guarda defensiva intencional para o caso em que a implementação de `loadTasks` mude no futuro.

```typescript
// Guarda defensiva: loadTasks suprime erros internamente e nunca lança,
// mas o catch preserva loading=false via finally mesmo em caso de mudança futura.
```

---

### Crítico 2 — JSON.parse sem validação de formato (CORRIGIDO)

**Arquivo:** `src/services/taskStorage.ts`, linha 13 (original)

**Problema:** `JSON.parse(data) as Task[]` — o `as Task[]` é um cast TypeScript que não existe em runtime. Se o storage contiver um JSON válido mas de formato diferente de array (ex: `"null"`, `{}`, `42`), o parse seria bem-sucedido e o valor seria propagado como `Task[]`, causando crash no primeiro `.map()` ou `.filter()` no contexto.

**CVE relevante:** Sem vetores externos (AppSec — o AsyncStorage é local), mas o crash por corrupção de dados é um risco de integridade de dados real.

**Correção aplicada:**

```typescript
const parsed: unknown = JSON.parse(data);
if (!Array.isArray(parsed)) return [];
return parsed as Task[];
```

### Importante 1 — updateTask não validava título (CORRIGIDO)

**Arquivo:** `src/context/TaskContext.tsx`, função `updateTask`

**Problema:** `addTask` validava o título (`if (!input.title.trim()) throw ...`), mas `updateTask` aceitava `{ title: '' }` silenciosamente, permitindo persistir tarefa com título vazio via chamada direta ao contexto (bypass da tela).

**Correção aplicada:**

```typescript
if (data.title !== undefined && !data.title.trim()) {
  throw new Error('Título é obrigatório');
}
```

O guard só dispara quando `title` é explicitamente passado (campo opcional no tipo).

---

### Importante 2 — updateTask e removeTask com ID inexistente causavam I/O desnecessário (CORRIGIDO)

**Arquivo:** `src/context/TaskContext.tsx`, funções `updateTask` e `removeTask`

**Problema:** Quando o ID não existia, as funções buildavam um `updated` idêntico ao `previous`, chamavam `setTasks` (re-render desnecessário) e `saveTasks` (write no AsyncStorage sem mudança real).

**Correção aplicada:**

```typescript
if (!tasksRef.current.some((t) => t.id === id)) return;
```

Adicionado antes do snapshot de `previous` em ambas as funções. No-op verdadeiro: sem I/O, sem re-render.

---

### Importante 3 — Rollback concorrente não documentado (DOCUMENTADO)

**Arquivo:** `src/context/TaskContext.tsx`, blocos catch dos mutadores

**Problema:** O rollback captura `previous` no momento da chamada. Em chamadas concorrentes sem `await` (ex: `addTask("A"); addTask("B")`), a falha de A pode sobrescrever o estado bem-sucedido de B, causando divergência UI/storage.

**Ação:** Adicionado comentário de aviso em cada bloco de rollback:

```typescript
// ATENÇÃO: rollback captura o estado no momento da chamada.
// Em mutações concorrentes sem await, o rollback desta falha pode sobrescrever
// o estado de outra mutação bem-sucedida. Evite múltiplas mutações simultâneas.
```

### Observação — Revisor identificou Minor #7 incorretamente

O revisor sugeriu remover `import type { TaskContextData }` de `useTasks.ts` como "redundante". A análise foi equivocada: o `export type { ... }` re-exporta o tipo para consumidores externos, mas **não coloca o tipo em escopo no arquivo atual**. A anotação de retorno `useTasks(): TaskContextData` exige o import local. A sugestão foi recusada com justificativa técnica.
