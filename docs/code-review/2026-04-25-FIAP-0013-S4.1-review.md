### 3.2 Importantes

#### I1 — Ausência de rollback nos métodos do TaskContext
**Arquivo:** `src/context/TaskContext.tsx:48-70` (pré-correção)

Os três métodos (`addTask`, `updateTask`, `removeTask`) atualizavam o estado em memória antes de chamar `saveTasks`, sem `try/catch`. Se `saveTasks` rejeitasse (storage cheio, AsyncStorage corrompido), a tela exibia os dados atualizados mas o AsyncStorage permanecia com o estado anterior. Na próxima abertura do app, o usuário perderia as alterações feitas na sessão.

**Correção aplicada (padrão otimista com rollback):**
```ts
async function addTask(input): Promise<void> {
  // ...
  const previous = tasksRef.current;
  const updated = [...previous, newTask];
  tasksRef.current = updated;
  setTasks(updated);
  try {
    await saveTasks(updated);
  } catch (error) {
    tasksRef.current = previous;
    setTasks(previous);
    throw error;
  }
}
```
Mesmo padrão aplicado em `updateTask` e `removeTask`.

#### I2 — Assinatura de `updateTask` permitia sobrescrita de campos imutáveis
**Arquivo:** `src/context/TaskContext.tsx:9` (interface)

A assinatura `updateTask(id: string, data: Partial<Task>)` permitia que o TypeScript aceitasse `{ id: 'outro-id' }` ou `{ createdAt: '...' }` sem erro de compilação, embora a implementação protegesse esses campos com `{ ...task, ...data, id, updatedAt }`. A intenção era comunicável via tipo.

**Correção aplicada:**
```ts
updateTask: (id: string, data: Omit<Partial<Task>, 'id' | 'createdAt' | 'updatedAt'>) => Promise<void>;
```

#### I3 — `TaskFormScreen` sem guarda de loading no modo edição
**Arquivo:** `src/screens/tasks/TaskFormScreen.tsx`

Se `useTasks()` ainda estivesse carregando quando a tela montasse em modo edição, `tasks.find(t => t.id === taskId)` retornaria `undefined` e todos os campos inicializariam vazios via `useState(existingTask?.title ?? '')`.

**Correção aplicada:**
```tsx
if (isEditing && loading) {
  return (
    <SafeAreaView style={styles.container}>
      <View style={styles.loadingContent}>
        <ActivityIndicator size="large" color="#007AFF" />
      </View>
    </SafeAreaView>
  );
}
```

## 4. Análise de Segurança e CVEs

### 4.1 AsyncStorage sem criptografia
`@react-native-async-storage/async-storage@2.2.0` persiste dados em texto plano no dispositivo (Android: SQLite sem encryption; iOS: arquivos sem NSFileProtection). Para um aplicativo educacional sem dados sensíveis (sem PII, sem tokens de autenticação nas tarefas), o risco é aceitável. Não há CVEs conhecidos relacionados a esta versão até a data desta auditoria.

### 4.2 `JSON.parse` sem validação de schema em `loadTasks`
**Arquivo:** `src/services/taskStorage.ts:13`

`return JSON.parse(data) as Task[]` é uma type assertion que não garante conformidade em runtime. Dados corrompidos ou de versões anteriores do app seriam retornados como `Task[]` sem erro de compilação. Para produção, seria necessária validação via zod ou similar.

**Risco:** Baixo para app educacional (dados locais, sem entrada remota).
**CVE:** Nenhum aplicável neste contexto.

### 4.3 `Math.random()` em `generateId`
IDs baseados em `Date.now() + Math.random()` não são criptograficamente seguros. Para um app educacional sem autenticação baseada em IDs, o risco é nulo. IDs gerados são usados apenas como chaves de lista local.

### 4.4 Injeção via campos de formulário
Os campos `title`, `description`, `category` e `categoryIcon` são serializados via `JSON.stringify` e lidos apenas pelo mesmo app. Sem superfície de ataque para XSS ou injection no contexto atual.
